# Gemini Cognitive Workspace & Caching Guidelines

본 문서는 대규모 컨텍스트 창(1M-2M+ 토큰) 및 서버사이드 컨텍스트 캐싱(Context Caching) 기능을 탑재한 Gemini 계열 모델(Gemini 3.5 Flash, 3.0 Flash, 3.1 Pro 등)에 맞추어 지식 베이스와 에이전트 워크플로우를 최적화하기 위한 설계 규칙을 정의합니다.

## 1. 컨텍스트 캐싱 최적화 프롬프트 설계 (Prompt Caching Rules)

Gemini의 컨텍스트 캐싱(최소 4,096 토큰 이상 활성화)은 프롬프트의 접두사(Prefix)가 100% 일치할 때 작동합니다. 접두사가 조금이라도 달라지면 캐시가 파기(Invalidation)되어 지연 시간과 비용이 상승합니다.

- **정적 지식 및 규칙의 최상단 배치**:
  - 시스템 규칙(System Prompts), 데이터 모델 스키마(Schemas), 프로젝트 스펙 및 아키텍처 문서 등 변하지 않는 정적 데이터는 프롬프트의 **최상단(Prefix)**에 영구 고정 배치합니다.
- **동적 변수의 최하단 배치 (접두사 보존)**:
  - 현재 시각(`current_time`), 세션 ID, 플레이어 입력 텍스트, 실시간 터미널/도구 실행 결과(Tool Outputs)와 같이 호출할 때마다 매번 변하는 동적 데이터는 반드시 프롬프트의 **최하단(Suffix)**에 추가해야 합니다.
  - 프롬프트 중간이나 상단에 동적 변수를 임베딩하는 규칙을 엄격하게 금지합니다.
- **캐시 경계성 유지 (Cache Boundary Control)**:
  - 서브 에이전트들은 각각 독립된 프롬프트 캐시 경계를 가져야 합니다. 한 에이전트의 도구 스키마가 업데이트된다고 해서 다른 에이전트의 캐시까지 무효화되지 않도록 시스템을 격리하여 설계합니다.

## 2. RAG를 대체하는 캐시 기반 생성 기법 (CAG: Cache-Augmented Generation)

전통적인 벡터 DB 기반 조각 검색(RAG) 대신, 1M+ 토큰 창에 전체 코드베이스 및 기획 사양서를 한 번에 적재하는 캐시 기반 생성(CAG)을 권장합니다.

- **CAG 적용 기준**:
  - 전체 파일 및 마크다운 사양서의 총 토큰 합계가 150만 토큰(약 1,000개 파일) 이하인 소형~중형 프로젝트의 경우, 실시간 RAG 대신 **전체 코드베이스와 스펙 문서를 캐시 영역에 사전 적재**하여 사용합니다.
  - 이를 통해 청크(Chunking) 유실 및 검색 순위 실패로 인한 인지 왜곡을 방지하고 에이전트에게 전체 아키텍처의 전역적 시야를 제공합니다.
- **캐시 프리워밍 (Cache Pre-warming)**:
  - 중요 에이전트 작동 전, 정적 프롬프트가 주입된 캐시(기본 1시간 유지)를 강제로 한번 호출하여 서버사이드 KV 캐시를 Warm 상태로 만들어 첫 쿼리의 prefill 지연 시간을 은닉합니다.

## 3. 다중 에이전트 위계 오케스트레이션 (Hierarchical Orchestration)

Gemini 모델들의 연산 속도와 추론 성능의 차이를 고려하여 역할을 위계적으로 나눕니다.

- **대뇌 모델 (Orchestrator)**:
  - **Gemini 3.1 Pro를 사용합니다.
  - 전역 스케줄링, 성찰(Reflection), 관계 감정 변동 계산 및 서브 태스크 분할 등 정교한 인지 추론을 담당합니다.
- **소뇌/척수 모델 (Workers)**:
  - **Gemini 3.5 Flash / 3.0 Flash**를 사용합니다.
  - 빠른 응답 속도(Pro 모델 대비 약 4배의 토큰 출력 성능)와 높은 도구 호출 정확도(Function Calling Accuracy)를 활용해 C++ 물리 명령 대기 연출, DB 로깅, 파일 입출력 등 단발성 실행 태스크를 전담합니다.
- **정보 병목 제어 (Information Bottleneck)**:
  - 서브 에이전트들의 실행 상세 로그를 Orchestrator에 그대로 전달하지 않습니다. 서브 에이전트 완료 시 핵심 결과 요약본만 추출하여 Orchestrator의 다음 인지 틱(Cognitive Tick)에 전달함으로써 컨텍스트 윈도우 포화 및 누적 비용을 방지합니다.

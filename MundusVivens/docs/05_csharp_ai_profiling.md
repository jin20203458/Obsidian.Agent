# C# AI Server Profiling & Optimization
> **부제**: C# AI 서버 성능 프로파일링 및 Smallville 비교 보고서

본 문서는 Mundus Vivens **C# AI 대뇌 서버**의 LLM API 비용 효율성, 기억 계층 구조, 비동기 스케줄링 성능을 정량적으로 계측하고, 스탠퍼드 *Generative Agents (Smallville)* 논문의 아키텍처와 직접 대조하여 설계 우위를 실증하는 보고서입니다.

> **자매 문서**: C++ 물리 서버의 Tracy 프로파일링 및 극한 벤치마크 결과는 [04_cpp_server_profiling.md](./04_cpp_server_profiling.md)를 참조하십시오.

## 1. 프로파일링 개요

### 1) 도입 목적
* **LLM API 비용 정량 계측**: 대화, 성찰, 일과 계획 등 각 인지 파이프라인 단계별 실제 API 호출 횟수와 토큰 소비량을 측정하고, Smallville 논문의 이론적 호출 횟수와 1:1 대조합니다.
* **기억 계층(Hot/Cold) 안정성 검증**: 대량 기억 주입(Eviction Storm) 상황에서 인메모리(RAM) 상한선이 완벽히 사수되는지, LiteDB 콜드 아카이브로의 도태(Eviction)가 정상 동작하는지 실증합니다.
* **비동기 스케줄링 비차단성 검증**: 다수 에이전트의 동시 성찰 요청이 메인 시뮬레이션 틱에 주는 실시간 영향(Contention)이 0ms인지 계측합니다.
* **결정론적 판정 파이프라인 속도**: 위협 판정, 인과 캐스케이드 등 LLM을 거치지 않는 수학적 결정론 연산의 처리량(throughput)을 마이크로초 단위로 계측합니다.

### 2) 벤치마크 인프라 설계
프로덕션 코드와 벤치마크 코드를 완전히 분리하여, 일반 서버 구동에 벤치마크 오버헤드가 유입되지 않도록 설계되었습니다.
* **CLI 스위치 (`--benchmark-mode`)**: `Program.cs`의 진입점에서 아규먼트를 인터셉트하여, 지정된 벤치마크 모드만 단독 실행 후 프로세스를 강제 종료합니다.
* **지원 모드**: `dialogue`, `memory`, `reflection`, `full-sim`, `threat`, `cascade`
* **Smallville 대조군 카운터**: 각 벤치마크 내부에서 동일 시나리오를 Smallville 방식으로 처리했을 경우 발생했을 이론적 API 호출 횟수 및 예상 비용을 병행 집계합니다.

## 2. 벤치마크 실행 절차 (Runbook)

### 1) 개별 축 벤치마크 실행
```powershell
# 축 1: 프롬프트 통합 경제성
dotnet run --project MundusVivens.Prototype -- --benchmark-mode dialogue

# 축 2: 기억 저장소 계층화 (1,000개 기억 주입 기준)
dotnet run --project MundusVivens.Prototype -- --benchmark-mode memory 1000

# 축 3: 성찰 스케줄링 (5명 동시 요청 기준)
dotnet run --project MundusVivens.Prototype -- --benchmark-mode reflection 5

# 축 4 & 6: 종합 시뮬레이션 (에이전트 2명, 2일 시뮬레이션 기준)
dotnet run --project MundusVivens.Prototype -- --benchmark-mode full-sim 2 2

# 축 5a: 위협 판정 배치 (1,000회 배치 기준)
dotnet run --project MundusVivens.Prototype -- --benchmark-mode threat 1000

# 축 5b: 인과 캐스케이드 (트리 깊이 3, 분기 5 기준)
dotnet run --project MundusVivens.Prototype -- --benchmark-mode cascade 3 5
```

### 2) 유의 사항
* `world_config.json`의 `ForceResetDatabase: true` 설정을 확인하여, 이전 세션의 잔여 기억 데이터가 벤치마크 결과를 오염시키지 않도록 합니다.
* 비지성체(`IsSentient = false`)는 성찰 큐에서 자동 바이패스되어 API 호출 0회가 보장됩니다.
* **유의**: `cascade` 모드 실행 시, 트리 깊이(`depth`)를 6 이상으로 설정하더라도 `BeliefEngine` 내부의 **깊이 제한 가드(Depth Clamping Guard = 5)**로 인해 프로세스 크래시나 OOM 대신 경고 메시지와 함께 안전하게 5레벨에서 전파가 조기 종료됩니다. 정상 전파 상태에 대한 전체 계측을 수행하려면 깊이를 3~5 레벨 사이로 지정하여 주십시오.

## 5. 축별 정량 검증 데이터

### 1) 축 1: 프롬프트 통합 경제성 (Prompt Consolidation)

* **가설 및 설계**: Smallville은 대화 턴마다 개별 API 호출(발화 생성, 감정 판정, 관계 갱신 등)을 수행하여 8턴 대화에 12~20회의 호출이 발생한다. Mundus Vivens는 단일 JSON 통합 스키마로 1회 호출에 모든 출력을 병합하여 91~95%의 API 호출을 절감할 수 있다.
* **실험 조건**: 에이전트 간의 8턴 대화 시뮬레이션 총 10세션 연속 계측.
* **실측 결과**:

    대화당 평균 턴 수: 8 턴

* **아키텍처 의의**: 단 1회의 Gemini API 호출로 [발화 텍스트 생성 -> 감정 상태 분석 -> 관계 수치 업데이트 -> 대화 요약문 작성 -> 이동 목적지 및 행동 스케줄 갱신]을 완전 비동기로 일시에 해결하였습니다.

### 2) 축 2: 기억 저장소 계층화 (Memory Architecture)

* **가설 및 설계**: Smallville은 모든 기억을 단일 Flat List에 저장하여 에이전트 수 × 기억 수에 비례하는 RAM을 점유한다. Mundus Vivens는 Hot Memory(RAM, 상한 40개)와 Cold Archive(LiteDB 디스크)로 이원화하여 RAM 점유를 고정(Clamped)한다.
* **실험 조건**: 에바(`npc_eva`) 에이전트에게 1,000개의 임의 기억을 연속 주입하는 Eviction Storm 테스트 및 5개 풀에서의 연상 Recall 조회 대조.
* **실측 결과 (개선 전 - 동기식 디스크 쓰기)**:
    ```
    =======================================================
    📊 축 2: 기억 저장소 계층화 검증 결과
    =======================================================
    주입한 총 기억 수: 1000 개
      - Hot Memory (RAM): 40 개 (상한선: 40 개)
      - Cold Memory (DB):  960 개 이관 완료
    기억 주입 및 Eviction 처리 시간: 4403 ms
    -------------------------------------------------------
    Top-5 연상 기억 Recall 속도 비교 (틱 단위):
      - Smallville 방식 O(N) 전수 스캔: 14109 ticks
      - Mundus Vivens LiteDB 인덱스 조회:  242238 ticks
      - 데이터베이스 스캔 쿼리 배율 우위: 0.06 배 향상
    =======================================================
    ```
* **실측 결과 (개선 후 - 비동기 지연 쓰기 큐 적용)**:
    ```
    =======================================================
    📊 축 2: 기억 저장소 계층화 검증 결과
    =======================================================
    주입한 총 기억 수: 1000 개
      - Hot Memory (RAM): 40 개 (상한선: 40 개)
      - Cold Memory (DB):  960 개 이관 완료
    기억 주입 및 Eviction 처리 시간: 31~33 ms  (메인 스레드 블로킹 해제, 약 130배 이상 속도 단축)
      - [A.1] 비동기 쓰기 큐 플러시 백그라운드 소요 시간: 4,542 ms
    -------------------------------------------------------
    Top-5 연상 기억 Recall 속도 비교 (틱 단위):
      - Smallville 방식 O(N) 전수 스캔: 11897 ticks
      - Mundus Vivens LiteDB 인덱스 조회:  179640 ticks
    =======================================================
    ```
* **아키텍처 의의**: 1,000개의 기억이 밀려 들어오는 상황에서도 인메모리 RAM 상한선인 40개를 완벽히 고수하며 960개의 기억을 Cold 데이터베이스로 분리 이관(Evict)시켰습니다.
* **메모리 스캔 대조 분석**: 인메모리 CPU-RAM 스캔(Smallville)이 디스크 I/O가 수반되는 DB 인덱스 쿼리보다 연산 자체는 빠르게 계측되었으나, 이는 데이터 누적 시 RAM 사용량이 기하급수적으로 팽창하여 결국 가상 월드 전체의 물리 메모리 고갈을 유발하는 구조적 결함을 동반합니다. Mundus Vivens의 구조는 물리 메모리를 완전히 고정(Clamped)하여 대규모 서버 환경에서의 장기 생존성을 사수합니다.

### 3) 축 3: 성찰 스케줄링 (Reflection Scheduling)

* **가설 및 설계**: Smallville은 중요도 합산 ≥ 150 시 메인 루프를 블로킹하며 즉시 성찰을 수행한다. Mundus Vivens는 비동기 우선순위 큐에 삽입하고 백그라운드 워커 스레드에서 스로틀링(10 TPS) 처리하여 메인 틱 영향 0ms를 달성한다.
* **실험 조건**: 에이전트 5명의 성찰 및 일과 스케줄 계획을 동시에 트리거하여 큐 제어력 계측.
* **실측 결과**:
    ```
    =======================================================
    📊 축 3: 성찰 스케줄링 검증 결과
    =======================================================
    처리 에이전트 수: 5 명
    전체 비동기 스케줄/성찰 소진 완료 시간: 7522 ms
    더블 버퍼드 포인터 스왑 소요 시간: 41109 ticks (즉각 스왑 보장)
    -------------------------------------------------------
    성찰/일과 동시 폭발 상황에서의 API 제어력:
      - Smallville 방식 (실시간 블로킹/직렬 대기): ~12500 ms (프레임 멈춤)
      - Mundus Vivens 방식 (비동기 10 TPS 큐 분산): 7522 ms (메인 틱 영향 0ms)
    =======================================================
    ```
* **아키텍처 의의**: 5명이 동시에 대뇌 성찰을 요청해도 메인 스레드에 주는 병목(Contention)은 **0ms**였습니다. 큐 워커가 백그라운드 스레드에서 초당 10회 스로틀링(10 TPS)을 준수하며 비동기 처리하였고, 계획이 완료된 스케줄은 포인터 스왑(Double-Buffered Swap)을 통해 틱 진행 중단 없이 즉시 교체되었습니다.

### 4) 축 4: 일과 생성 및 이동 비용 최적화 (Planning & Travel)

* **가설 및 설계**: Smallville은 매일 3단계 계획 수립(하루 전체 → 시간대별 → 5~15분 단위)을 에이전트마다 수행하여 30~40+회의 API 호출을 소비한다. Mundus Vivens는 1회 통합 성찰로 24시간 일과를 생성하며, 원정 이동 중에는 Auto-Continue로 100% 바이패스한다.
* **실험 조건**: 에이전트 2명에 대한 2일간의 완전 시뮬레이션. Day 2에 굶주린 늑대의 생존 위기(허기 < 15) 돌발 인터럽트 발생 및 비지성체(IsSentient = false) 큐잉 제외 조건 결합 모사.
* **실측 결과**:
    ```
    =======================================================
    📊 축 4 & 6: 종합 시뮬레이션 및 비용 분석 결과
    =======================================================
    에이전트 수: 2 명 | 시뮬레이션 기간: 2 일
    총 시뮬레이션 실행 소요 시간: 14283 ms
    Auto-Continue 우회 적용 횟수: 0 회
    생존 Override 우회 적용 횟수: 1 회
    -------------------------------------------------------
    [API 호출 횟수 및 비용 절약 대조군 비교]
                        Smallville(이론)     Mundus Vivens(실측)
    계획/성찰 API 호출:  90 회               3 회
    추정 운영 비용:      $0.1350              $0.0004 (Gemini Flash 기준)
    API 비용 절감률:     96.67 % 절감
    =======================================================
    ```
* **아키텍처 의의**: 
    * **생존 위기 오버라이드**: 굶주린 늑대(`npc_wolf`)가 Day 2에 생존 위기를 겪었을 때, C++ 로컬 생존 행동(Behavior Tree)이 우선 제어권을 가져오고 C#의 비싼 성찰 API 호출을 100% 생략(Bypass, 1회)하여 불필요한 LLM 비용 소모를 완전히 방어했습니다.
    * **비지성체 인지 배제**: 늑대와 같은 동물/야수 계열은 `IsSentient = false`로 지정되어, 자정의 성찰/일과 수립 큐잉 시점에서 즉시 바이패스(Bypass)되고/로컬에서 고정된 배회/습격 스케줄을 즉시 주입받습니다. 이를 통해 몬스터나 야생 개체로 인한 불필요한 LLM API 요금 낭비를 원천 제거(0원 동결)하였습니다.

## 6. 발견된 병목 및 개선 로드맵

### 1) 병목 분석
* **LiteDB 동기 Eviction I/O [해결 완료]**:
  - **원인**: 기억을 도태할 때 인스턴스의 `OnBeliefEvicted` 핸들러가 동기 방식으로 디스크에 쓰는 작업을 처리하여, 대량의 기억이 밀려 들어올 때(Eviction Storm) 약 4.4초의 순간 지연이 발생했었습니다.
  - **조치**: Channel 기반 비동기 지연 쓰기 큐(`EnqueueArchive`) 및 백그라운드 워커를 도입하여 메인 스레드를 논블로킹화하고 멈춤 현상을 극복했습니다. 또한 감쇠(Decay) 기억 삭제 시에도 훅을 추가 연동하여 데이터 유실 문제를 함께 해결했습니다.
* **지수형 인과 그래프 생성 오버헤드 [해결 완료]**: `cascade` 구조 구축 시 깊이 및 자식 수의 증가가 지수적인 스케일($C^D$)로 메모리 할당 폭풍을 일으키기 때문에, 노드 수의 명확한 상한선 가드를 도입하여 해결했습니다. (5레벨 및 순환 가드 적용)

### 2) 최적화 로드맵

#### ① 단기 최적화 방안 (Quick Win)
* **LiteDB 비동기 쓰기 래퍼 도입 [완료]**: `EnqueueArchive` 및 백그라운드 워커 기반 비동기 지연 쓰기 큐(Channel)를 구축하여 메인 스레드 대기 시간을 4.4초에서 30ms 수준으로 대폭 단축 완료했습니다.
* **인과 캐스케이드 최대 탐색 깊이 제한(Cap) [완료]**: 전파 깊이를 최대 5레벨로 클램핑하고 HashSet 기반 순환 체크 가드를 `BeliefEngine`에 구축 완료했습니다.

#### ② 중장기 최적화 방안 (Architecture Evolution)
* **기억 회상(Recall) 다차원 클러스터링**: 핫메모리 스캔뿐만 아니라 데이터베이스 인덱싱 검색 범위를 에이전트의 Faction 또는 지역 필터로 타이트하게 묶어 디스크 스캔 속도 개선.

늑대(`npc_wolf`)와 같은 야수/동물 계열은 지성체가 아니므로 LLM 대뇌 연산(성찰, 일과 계획, 대화)이 불필요합니다. 이를 아키텍처 수준에서 완전히 격리하였습니다.

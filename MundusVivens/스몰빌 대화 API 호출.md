Searched web: ""Generative Agents" Stanford conversation dialogue prompt turn by turn api calls"

스탠퍼드 **스몰빌(Stanford Generative Agents)의 공식 깃허브 오픈소스 코드와 논문 명세를 기반으로 확인한 확실한 정보**로 답변드립니다.

스몰빌에서 두 NPC가 만나 **8턴짜리 대화(A가 4번, B가 4번 발화)**를 나누고 이를 기억에 기록하는 과정까지 발생하는 총 API 호출 수는 **최소 12회에서 최대 20회**입니다. 

말씀해주신 시나리오대로 대화 한 번에 아래와 같이 정직하게 턴 단위로 쪼개져 엄청난 횟수의 API 호출이 일어납니다.

---

### 1. 단계별 실제 API 호출 흐름 (8턴 대화 기준)

#### ① 대화 여부 판정 (Reaction Decision) ➔ 총 2회 호출
* NPC A가 NPC B를 시야에서 발견하고 대화할지 여부(`should_react`)를 LLM에 물어봅니다. **(1회)**
* NPC B도 자신의 업데이트 틱에서 NPC A를 발견하고 대화할지 여부를 판정합니다. **(1회)**

#### ② 실제 대화 생성 (Turn-by-Turn Dialogue) ➔ 총 8회 호출 (턴 수에 비례)
스몰빌은 대본 전체를 한 번에 쓰지 않고, 진짜 대화처럼 **상대 대사를 보고 내가 대답하는 턴제 구조**로 작동합니다.
* NPC A의 첫 대사 생성 **(1회)**
* NPC B가 A의 첫 대사를 보고 자신의 다음 대사 생성 **(1회)**
* NPC A가 B의 대사를 보고 대사 생성 **(1회)**
* ... (이를 주고받으며 `<goodbye>` 태그가 나오거나 한도에 도달할 때까지 반복. 보통 8턴 기준 **총 8회 호출**)

#### ③ 대화 사후 요약 (Summarization) ➔ 총 2회 호출
* 대화가 끝나면 NPC A는 자신의 입장에서 전체 대본을 보고 "이번 대화의 팩트 요약본"을 생성해 기억에 기록합니다. **(1회)**
* NPC B도 자신의 입장에서 대화 대본을 요약하여 기억에 기록합니다. **(1회)**

#### ④ 사후 성찰 (Reflection) ➔ 조건 충족 시 총 8회 추가 (A: 4회 / B: 4회)
스몰빌은 매 대화 직후 성찰하지 않고, 새로 저장된 기억의 중요도 누적 합산이 **150점**을 넘는 순간 성찰이 켜집니다. 대화 요약본이 들어오면서 150점이 넘어가면 다음 호출이 수행됩니다.
* NPC A가 최근 기억을 보고 "궁금한 질문 3개 추출" **(1회)** ➔ 질문 3개 각각에 대해 "과거 기억을 회상해 통찰/자아 도출" **(3회)**
* NPC B도 똑같이 질문 추출 **(1회)** ➔ 통찰 도출 **(3회)**

---

### 📊 8턴 대화 1회당 API 호출 총합
* **성찰이 켜지지 않은 일반 대화**: **12회 API 호출** (판정 2회 + 대화 8회 + 요약 2회)
* **성찰이 켜진 대화**: **20회 API 호출** (성찰 8회 추가)

이처럼 무지막지하게 LLM을 호출해댔기 때문에, 스몰빌 프로젝트는 동시 접속자 25명이 단 하루 동안 시뮬레이션을 돌리는 데 **수천 달러(수백만 원)의 API 요금**이 나오는 치명적인 비용 실패를 겪었습니다. 

우리가 대화 턴마다 LLM을 호출하지 않고, **한 번의 호출로 전체 대화 대본 세션과 감정 변화를 동시에 받아내는 오케스트레이터 구조**로 최적화한 이유가 바로 이 스몰빌의 비효율성을 극복하기 위함이었습니다.


- **M**(Modified): 나만 수정함 (최신)
    
- **E**(Exclusive): 나만 깔끔하게 들고 있음
    
- **S**(Shared): 너랑 나랑 같이 들고 있음
    
- **I**(Invalid): 내 캐시에 있는 건 옛날 쓰레기임
[[Invalidation Queue]]

**MESI 프로토콜은 서로 다른 캐시의 동기화가 목적이라 L1, L2는 사실상 동일 취급**

### 1. 수정 과정 (S ➔ I ➔ E ➔ M)

> _"다른 놈들 I로 만들고 자기가 E가 된 데이터를 수정해서 M"_

**[정확함]** 공유 중(S)인 데이터를 고치려면 반드시 남들의 데이터를 무효화(I) 시켜서 내 캐시에만 유일하게 남는 독점 상태(E)를 만들어야 합니다. 방해꾼이 없는 이 E 상태가 확보된 직후에야 비로소 값을 고치고 수정됨(M) 상태로 마킹할 수 있습니다.

### 2. 전파 과정 (M ➔ S)

> _"다른 코어가 저 데이터를 요청하면 저걸 주고 다시 둘 다 S로"_

**[정확함]** 내가 우주 최신본(M)을 들고 있는데 다른 코어가 낡은 원본(RAM)을 읽으려고 하면, 내가 잽싸게 끼어들어 내 최신 값을 복사해 넘겨줍니다. 이제 둘 다 똑같은 최신 값을 나눠 가졌으므로, 내 상태는 M에서 S(공유)로 강등되고 상대방도 S 상태가 되어 다시 평화로운 동기화 상태가 됩니다.

### 3. 방출 과정 (Write-Back)

> _"메모리는 각 데이터가 밀릴 때 실제 메모리에 쓰기가 되는 거 맞지?"_

**[정확함]** CPU는 RAM을 극도로 혐오할 만큼 기피합니다. M 상태의 데이터라 할지라도 캐시 메모리 공간이 꽉 차서 **어쩔 수 없이 방을 빼야 할 때(Eviction, 밀릴 때)** 비로소 L2, L3를 거쳐 최종적으로 실제 메인 메모리(RAM)에 쓰기(Write)가 발생합니다.

최근 아키텍처는 순수 MESI보다는 이를 개선한 확장 프로토콜인 **MESIF**와 **MOESI**를 주로 사용하며, 제조사(Intel, AMD, ARM)에 따라 뚜렷하게 갈립니다.

## 1. 최근 아키텍처별 프로토콜 채택 현황

현대 CPU 시장을 이끄는 주요 제조사들은 각각 자신들의 버스 구조와 칩 설계(단일 다이 vs 칩렛 구조)에 최적화된 프로토콜을 사용하고 있습니다.

|**제조사**|**사용하는 프로토콜**|**특징**|
|---|---|---|
|**Intel (인텔)**|**MESIF**|기존 MESI에 **F(Forward)** 상태를 추가.|
|**AMD**|**MOESI**|기존 MESI에 **O(Owned)** 상태를 추가.|
|**ARM**|**MOESI / MESI**|스마트폰 등에 쓰이는 Cortex-A 시리즈는 대개 MOESI(또는 커스텀 MESI 계열) 사용.|

## 2. 왜 순수 MESI 대신 이걸 쓸까? (확장 이유)

순수 MESI는 코어가 많아질수록 불필요한 메모리 쓰기(Write-back)나 **중복 버스 트래픽**이 발생해 성능 병목이 생깁니다. 이를 해결하기 위해 상태를 하나씩 더 추가한 것입니다.

### 인텔의 MESIF : "응답자 지정으로 트래픽 최소화"

여러 코어가 같은 데이터를 읽어서 전부 **S(Shared)** 상태로 가지고 있을 때가 문제입니다. 이 상태에서 또 다른 코어가 "나도 그 데이터 읽을래!" 하고 요청하면, S 상태를 가진 수많은 코어들이 동시에 데이터를 던져주려고 버스(Interconnect)가 터져나가거나, 결국 느린 메인 메모리에서 가져와야 했습니다.

- **F(Forward) 상태의 도입:** 데이터를 공유할 때 딱 하나의 코어에게만 **F(Forward)** 마크를 붙여줍니다.
    
- 앞으로 누군가 같은 데이터를 요청하면 **오직 F 상태를 가진 코어만 캐시-투-캐시 전송으로 응답**합니다. 메인 메모리까지 갈 필요도 없고, 다른 S 상태 코어들은 조용히 하던 일만 하면 되므로 트래픽이 획기적으로 줄어듭니다.
    

### AMD의 MOESI : "메모리 쓰기를 최대한 지연"

순수 MESI에서는 내가 **M(Modified)** 상태로 수정한 데이터를 다른 코어에게 넘겨줄 때, 강제로 메인 메모리(또는 L3)에도 동시에 값을 써야(Write-back) 했습니다. 그래야 둘 다 **S(Shared)** 상태가 될 수 있으니까요. 하지만 메모리에 쓰는 작업은 너무 느립니다.

- **O(Owned) 상태의 도입:** 내가 수정한 데이터를 메모리에 쓰지 않은 채로 다른 코어에게 넘겨줍니다. 대신 나는 **O(Owned)** 상태가 되고, 데이터를 받은 코어는 **S(Shared)** 상태가 됩니다.
    
- 이 상태는 *"메모리 값은 아직 옛날 값(Dirty)이지만, 최신 데이터는 우리끼리 공유하고 있다. 나중에 내가 캐시에서 쫓겨날 때 메모리에 쓸 테니 일단 메모리 쓰기는 미루자!"*라는 뜻입니다. 덕분에 느린 메모리 접근 횟수가 대폭 줄어듭니다. 특히 AMD는 여러 칩을 이어 붙이는 **칩렛(Chiplet) 구조**를 쓰기 때문에 메모리 접근을 미루는 이 방식이 아주 유리합니다.
    

## 3. 또 하나의 숨은 핵심: Snoop Filter (디렉터리 방식)

여기에 현대 멀티코어 CPU(코어가 수십 개에 달하는 서버용 제품 등)는 무작정 버스에 신호를 흩뿌리는 스누핑만 쓰지 않습니다.

대신 중간에 **스눕 필터(Snoop Filter)** 혹은 디렉터리(Directory)라는 하드웨어 추적기를 둡니다. 이 장치가 _"어떤 코어가 어떤 데이터 라인을 무슨 상태로 들고 있는지"_ 미리 목록으로 쥐고 있습니다. 그래서 다른 코어가 요청을 보내면 버스 전체에 소리 지르는 대신, **그 데이터를 진짜 가지고 있는 코어에게만 콕 집어서(Unicast)** 요청을 전달합니다.

> **요약**
> 
> 최근 CPU 아키텍처는 **인텔은 MESIF, AMD/ARM은 MOESI**를 기반으로 작동하며, 코어 수가 많아진 현대 환경에 맞춰 **스눕 필터(디렉터리 구조)**를 결합해 버스 트래픽과 메모리 병목을 억제하는 형태로 발전했습니다.
